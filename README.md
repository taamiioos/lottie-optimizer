# Lottie Optimizer

- **Вход** — Lottie JSON с base64-изображениями внутри
- **Выход** — оптимизированный JSON со ссылками на файлы плюс `assets.zip` с видео и картинками

---

## Как использовать

1. Откройте `optimizer.html` и перетащите `.json` файл в зону загрузки
2. Дождитесь завершения оптимизации
3. Скачайте `optimized.json` и `assets.zip`
4. Откройте `player.html`, загрузите оба файла и нажмите «Воспроизвести»

---

## Оптимизатор: этапы работы

### Этап 1 — Анализ исходного файла

- Оптимизатор проходит по всем ассетам в массиве `assets` и отбирает те, у которых поле `p` содержит Data URL с base64-данными изображения
- Для каждого такого ассета декодируется base64 в байты, определяется формат и фиксируется размер
- Формат определяется не по MIME-типу из Data URL, а по первым байтам самого файла — так называемым magic bytes. Это надёжнее, потому что MIME мог быть задан неверно
- По итогам анализа известно: сколько всего изображений, какого они формата, каков суммарный размер до оптимизации

---

### Этап 2 — Поиск последовательностей кадров

**Зачем это нужно?** Покадровая анимация — это набор PNG-изображений, снятых последовательно. Видеокодек сжимает такой набор в разы эффективнее, чем отдельные изображения, потому что кодирует только разницу между соседними кадрами, а не каждый кадр целиком.

**Как работает поиск?**
- Каждый ассет имеет поле `id`, по которому можно определить, является ли он частью серии
- Оптимизатор проверяет `id` на четыре паттерна:
  - Чисто числовой id — например `0`, `1`, `123`. Такой формат редко встречается, но используется в некоторых экспортёрах
  - Id с UUID-суффиксом — например `frame_0001_a3f8b2c1`. Это формат, который выдают After Effects-плагины при автоматическом именовании
  - Id с буквенным префиксом и числом через разделитель — например `image_001` или `frame-42`. Самый распространённый человекочитаемый формат
  - Id без разделителя — например `img0`, `frame12`. Упрощённый вариант, часто встречается в анимациях, созданных вручную
- Ассеты с одинаковым текстовым префиксом и разными числами образуют группу
- Группа считается последовательностью только если в ней три и более кадров — меньше трёх нет смысла кодировать в видео, потому что накладные расходы MP4-контейнера и обязательного ключевого кадра перевесят выигрыш от сжатия

---

### Этап 3 — Кодирование в H.264

Это наиболее универсальный видеокодек с аппаратным декодированием на всех современных устройствах. Для покадровых иллюстраций, где соседние кадры мало отличаются друг от друга, сжатие в 5–15 раз по сравнению с PNG — обычный результат.

**Как кодируется каждая последовательность.**
- Каждый кадр рисуется в невидимый OffscreenCanvas — это позволяет работать вне основного потока браузера и не блокировать интерфейс
- Из Canvas создаётся VideoFrame с указанием временной метки — она вычисляется как номер кадра, делённый на частоту кадров
- VideoFrame передаётся в VideoEncoder — браузерный API для аппаратного кодирования видео
- Каждый 30-й кадр принудительно делается ключевым. Ключевые кадры кодируются целиком, без опоры на соседние — это нужно плееру, который может запросить произвольный кадр анимации в любой момент
- После кодирования все чанки упаковываются в MP4-контейнер через библиотеку MP4Box
- Если итоговый MP4 оказался тяжелее суммы оригинальных кадров — видео отбрасывается и кадры обрабатываются как обычные изображения

**Выбор уровня H.264.** Стандарт делит кодек на уровни по сложности декодирования — декодер обязан поддерживать уровень, указанный в файле. Уровень выбирается по количеству макроблоков в кадре, чтобы взять минимально достаточный

**GPU-ускорение.** Перед началом кодирования проверяется, поддерживает ли браузер аппаратное кодирование для данного разрешения и кодека. Если да — используется GPU-энкодер, если нет — программный.

**Выравнивание размеров.** H.264 требует чётных значений ширины и высоты. Если исходное изображение нечётное по одному из измерений — добавляется один пиксель.

---

### Этап 4 — Обработка изображений

Ассеты, не вошедшие в видеопоследовательности, обрабатываются по одному.

**Дедупликация.**
- Для байт каждого изображения вычисляется SHA-256 хеш
- Если такой хеш уже встречался — изображение является дубликатом. Вместо сохранения нового файла ассет просто получает ссылку на уже существующий
- Это важно для анимаций, где одно и то же изображение используется в нескольких слоях: оно хранится в ZIP единственный раз

**WebP-конвертация.**
- Для уникальных изображений производится попытка перевода в WebP
- Браузер рисует картинку в скрытый Canvas и вызывает `toBlob` с форматом WebP и качеством 80%
- Качество 80% выбрано как разумный компромисс: WebP при таком уровне обычно в 1.5–2 раза меньше PNG при визуально неразличимом качестве для иллюстративной графики
- Если браузер не поддерживает WebP, или если получившийся файл оказался тяжелее оригинала — сохраняется оригинальный формат. Это важно для уже сжатых изображений, например JPEG-фотографий, где WebP не даст выигрыша
- Поддержка WebP проверяется один раз при запуске и кэшируется

**Именование файлов.**
- Имя файла строится из первых 8 символов SHA-256 хеша
- Это обеспечивает уникальность без коллизий на типичных объёмах Lottie-анимаций и одновременно реализует дедупликацию — два одинаковых изображения получат одно и то же имя и перезапишут друг друга

---

### Этап 5 — Упаковка в ZIP

- Все видео и изображения собираются в архив через JSZip
- Используется метод хранения без дополнительного сжатия. MP4 и WebP уже содержат собственные алгоритмы компрессии, повторное применение Deflate не дало бы выигрыша, но потратило бы время
- Видео кладутся в папку `video/`, изображения — в `assets/`
- В `optimized.json` у каждого ассета-изображения поля `u` и `p` обновляются до пути к файлу в архиве. У ассетов-кадров поля очищаются, а в JSON верхнего уровня добавляется массив `videoAssets` с метаданными каждого видео: путь к файлу, разрешение, количество кадров, частота кадров и список `frameIds` — оригинальных id ассетов в правильном порядке

## Плеер: этапы работы

### Этап 1 — Загрузка и разбор файлов

- Пользователь загружает `optimized.json` и `assets.zip`
- JSON парсится сразу в памяти
- ZIP разбирается через JSZip — все файлы остаются в памяти, готовые к извлечению по пути
- Кнопка «Воспроизвести» заблокирована до тех пор, пока оба файла не загружены

---

### Этап 2 — Извлечение изображений

- Плеер проходит по всем ассетам в JSON и отбирает те, у которых есть путь к файлу в архиве
- Для каждого такого ассета путь склеивается из полей `u` и `p`
- Файл извлекается из ZIP как Blob и превращается в временный `blob://` URL
- Этот URL подставляется прямо в поле `p` ассета
- После подмены ассет выглядит для bodymovin как обычная ссылка на изображение — библиотека не знает ни о каком ZIP

---

### Этап 3 — Декодирование видео

bodymovin работает с изображениями, а не с видеопотоком. Он не умеет извлекать кадр из видео в нужный момент времени — ему нужна готовая картинка для каждого кадра анимации. Поэтому плеер декодирует весь MP4 заранее и превращает каждый кадр в отдельный PNG с `blob://` URL.

**Основной метод — WebCodecs API.**

- Сначала MP4Box разбирает контейнер и выделяет все закодированные сэмплы видеодорожки
- Из внутренней структуры MP4 извлекается AVCDecoderConfigurationRecord — блок байт, описывающий параметры кодека: профиль, уровень, SPS и PPS. Без этого блока VideoDecoder не может начать декодирование H.264
- Перед созданием декодера проверяется доступность GPU-ускорения для данного разрешения и кодека
- Создаётся VideoDecoder, которому передаются все сэмплы по очереди. Каждый сэмпл помечается как ключевой или разностный — это нужно декодеру для правильной сборки изображения из дельт
- По мере декодирования каждый кадр попадает в callback, где рисуется в OffscreenCanvas и конвертируется в PNG Blob
- Конвертация всех кадров происходит параллельно через Promise.all — не нужно ждать окончания одного перед началом следующего

**Почему PNG, а не что-то другое.** После H.264-декодирования кадр — это просто растровая картинка в памяти. PNG здесь служит только оберткой для передачи её через `blob://` URL в bodymovin. Дополнительной потери качества нет — данные уже прошли через H.264-компрессию при оптимизации, PNG лишь сохраняет декодированный результат без изменений.

**Резервный метод — через элемент `<video>`.**

- Создаётся скрытый элемент `<video>`, которому передаётся Blob с видеофайлом
- После загрузки метаданных для каждого кадра устанавливается `currentTime` равный нужному моменту времени
- После каждой перемотки плеер ждёт событие `seeked`, затем рисует текущий кадр на Canvas и конвертирует в PNG
- Метод работает везде, но значительно медленнее: каждая перемотка — синхронная операция, GPU-ускорения нет

---

### Этап 4 — Сборка и запуск анимации

- После декодирования blob URL каждого кадра подставляется обратно в нужный ассет по его `id`
- Порядок кадров восстанавливается точно, потому что в `optimized.json` сохранён массив `frameIds` — оригинальные id ассетов в правильном порядке, привязанные к каждому видеофайлу
- Когда все ассеты содержат прямые ссылки, вызывается `lottie.loadAnimation()` с canvas-рендерером
- bodymovin получает обычный Lottie JSON с blob URL вместо base64 и воспроизводит анимацию штатным образом — он не знает ни о каком ZIP, ни о каком видео
